# Документ требований

## Введение

Fabric мод "cuffsx" для Minecraft 1.20.1, добавляющий механику наручников. Два предмета — наручники на руки и наручники на ноги — надеваются на других игроков через ПКМ. Игрок в наручниках на руках лишается всех взаимодействий с миром; игрок в наручниках на ногах не может двигаться. Состояние сохраняется через PersistentState и переживает рестарт сервера. Управление через команды `/cuffsx` с LuckPerms-пермишонами.

## Глоссарий

- **Мод**: мод cuffsx, mod id `cuffsx`, пакет `org.gloyer057.cuffsx`
- **HandcuffsItem**: предмет наручников (`cuffsx:handcuffs_hands` илиcuffsx:handcuffs_legs`)
- **CuffedPlayer**: игрок, на которого надеты наручники (хранится в CuffState)
- **Applier**: игрок, который надевает или снимает наручники
- **CuffState**: серверное состояние (PersistentState), хранящее список CuffedPlayer с метаданными
- **CuffType**: тип наручников — `HANDS` (руки) или `LEGS` (ноги)
- **CuffRecord**: запись в CuffState: цель, тип, кто надел, время, координаты
- **LuckPerms**: плагин/мод управления правами; пермишоны проверяются через Fabric Permissions API
- **Validator**: компонент, проверяющий, разрешено ли надевание (глобальный флаг enabled)

## Требования

---

### Требование 1: Предметы наручников

**Пользовательская история:** Как администратор сервера, я хочу, чтобы предметы наручников существовали в игре, чтобы я мог выдавать их игрокам через `/give`, или же в меню креатива, то есть нужна отдельная вкладка cuffsx где будут оба вида наручников.

#### Критерии приёмки

1. МОД ДОЛЖЕН зарегистрировать предмет `cuffsx:handcuffs_hands` с максимальным стеком 16.
2. МОД ДОЛЖЕН зарегистрировать предмет `cuffsx:handcuffs_legs` с максимальным стеком 16.
3. МОД ДОЛЖЕН предоставить JSON-модели предметов для `handcuffs_hands` и `handcuffs_legs` в `assets/cuffsx/models/item/`.
4. МОД ДОЛЖЕН предоставить файлы текстур-заглушек для `handcuffs_hands` и `handcuffs_legs` в `assets/cuffsx/textures/item/`.
5. МОД ДОЛЖЕН предоставить записи локализации на английском и русском языках для обоих предметов в `assets/cuffsx/lang/`.

---

### Требование 2: Надевание наручников через ПКМ

**Пользовательская история:** Как игрок с наручниками в руке, я хочу нажать ПКМ по другому игроку, чтобы надеть на него наручники и ограничить его.

#### Критерии приёмки

1. КОГДА игрок нажимает ПКМ по другому игроку, держа в руке `cuffsx:handcuffs_hands`, МОД ДОЛЖЕН надеть наручники типа `HANDS` на целевого игрока.
2. КОГДА игрок нажимает ПКМ по другому игроку, держа в руке `cuffsx:handcuffs_legs`, МОД ДОЛЖЕН надеть наручники типа `LEGS` на целевого игрока.
3. КОГДА наручники успешно надеты, МОД ДОЛЖЕН уменьшить размер стека HandcuffsItem на 1.
4. КОГДА наручники надеты, МОД ДОЛЖЕН создать CuffRecord, содержащий: UUID цели, UUID надевшего, CuffType, серверную метку времени и блочные координаты цели в момент надевания.
5. КОГДА наручники надеты, МОД ДОЛЖЕН сохранить CuffRecord в CuffState.
6. ЕСЛИ на целевом игроке уже надет тот же тип наручников (CuffType), МОД НЕ ДОЛЖЕН надевать дубликат и НЕ ДОЛЖЕН расходовать предмет.
7. ЕСЛИ глобальный флаг Validator отключён, МОД НЕ ДОЛЖЕН надевать наручники и ДОЛЖЕН отправить сообщение "Наручники нельзя одеть на игрока в данный момент." надевающему (Applier).
8. МОД ДОЛЖЕН надевать наручники ТОЛЬКО через взаимодействие ПКМ; команды НЕ ДОЛЖНЫ надевать наручники.

---

### Требование 3: Снятие наручников через ПКМ

**Пользовательская история:** Как свободный игрок, я хочу нажать ПКМ по игроку в наручниках, чтобы снять с него наручники и освободить его.

#### Критерии приёмки

1. КОГДА игрок, на котором нет наручников ни одного типа, нажимает ПКМ по CuffedPlayer, МОД ДОЛЖЕН снять один тип наручников с цели, отдавая приоритет `HANDS` перед `LEGS`.
2. КОГДА наручники сняты, МОД ДОЛЖЕН добавить соответствующий предмет HandcuffsItem в инвентарь снявшего (Applier).
3. КОГДА наручники сняты, МОД ДОЛЖЕН удалить соответствующий CuffRecord из CuffState.
4. ЕСЛИ на самом Applier надет любой тип наручников, МОД НЕ ДОЛЖЕН позволять ему снимать наручники с других.
5. ЕСЛИ на целевом игроке нет наручников, МОД НЕ ДОЛЖЕН выполнять никаких действий по снятию при нажатии ПКМ.

---

### Требование 4: Ограничения для игрока в наручниках на руках

**Пользовательская история:** Как администратор сервера, я хочу, чтобы игрок в наручниках на руках не мог взаимодействовать с миром, чтобы он был полностью обездвижен.

#### Критерии приёмки

1. ПОКА на игроке надеты наручники `HANDS`, МОД ДОЛЖЕН отменять все попытки сломать блок этим игроком.
2. ПОКА на игроке надеты наручники `HANDS`, МОД ДОЛЖЕН отменять все попытки поставить блок этим игроком.
3. ПОКА на игроке надеты наручники `HANDS`, МОД ДОЛЖЕН отменять все попытки взаимодействия с блоками (сундуки, двери, люки и т.д.) этим игроком.
4. ПОКА на игроке надеты наручники `HANDS`, МОД ДОЛЖЕН отменять все попытки использовать предметы этим игроком.
5. ПОКА на игроке надеты наручники `HANDS`, МОД ДОЛЖЕН отменять все попытки атаковать игроков и мобов этим игроком.
5. ПОКА на игроке надеты наручники `HANDS`, МОД ДОЛЖЕН отменять все попытки перемещать предметы в инвентаре/как либо взаимодействовать с ними.

---

### Требование 5: Ограничения для игрока в наручниках на ногах

**Пользовательская история:** Как администратор сервера, я хочу, чтобы игрок в наручниках на ногах не мог двигаться и оставался на месте.

#### Критерии приёмки

1. ПОКА на игроке надеты наручники `LEGS`, МОД ДОЛЖЕН установить атрибут скорости передвижения игрока в 0, запрещая ходьбу и бег.
2. ПОКА на игроке надеты наручники `LEGS`, МОД ДОЛЖЕН отменять все попытки прыжка этим игроком.
3. ПОКА на игроке надеты наручники `LEGS`, МОД ДОЛЖЕН телепортировать игрока обратно на позицию, зафиксированную в момент надевания наручников, если позиция игрока изменилась более чем на 0.1 блока за тик.

---

### Требование 6: Сохранение состояния (PersistentState)

**Пользовательская история:** Как администратор сервера, я хочу, чтобы состояние наручников сохранялось после перезапуска сервера, чтобы игроки не могли сбежать из-под стражи через перелогин.

#### Критерии приёмки

1. CuffState ДОЛЖЕН расширять `PersistentState` и храниться в директории данных мира сервера под ключом `cuffsx_state`.
2. КОГДА сервер сохраняет данные мира, CuffState ДОЛЖЕН сериализовать все активные CuffRecord в NBT.
3. КОГДА сервер загружает данные мира, CuffState ДОЛЖЕН десериализовать все CuffRecord из NBT и восстановить список активных наручников.
4. КОГДА игрок заходит на сервер, МОД ДОЛЖЕН повторно применить все ограничения движения/взаимодействия для этого игрока, если CuffState содержит активные CuffRecord для его UUID.
5. ДЛЯ ВСЕХ объектов CuffRecord сериализация с последующей десериализацией ДОЛЖНА давать эквивалентный CuffRecord (свойство round-trip).

---

### Требование 7: Команда `/cuffsx reload`

**Пользовательская история:** Как администратор сервера, я хочу перезагружать конфигурацию мода без перезапуска сервера, чтобы изменения вступали в силу немедленно.

#### Критерии приёмки

1. КОГДА игрок с пермишоном `cuffsx.reload` выполняет `/cuffsx reload`, МОД ДОЛЖЕН перезагрузить конфигурацию и отправить подтверждающее сообщение исполнителю.
2. ЕСЛИ игрок без пермишона `cuffsx.reload` выполняет `/cuffsx reload`, МОД ДОЛЖЕН отклонить команду и отправить сообщение об отсутствии прав.

---

### Требование 8: Команда `/cuffsx list`

**Пользовательская история:** Как администратор сервера, я хочу видеть список всех игроков в наручниках, чтобы отслеживать состояние сервера.

#### Критерии приёмки

1. КОГДА игрок с пермишоном `cuffsx.list` выполняет `/cuffsx list`, МОД ДОЛЖЕН отобразить список всех активных CuffRecord, включая: имя целевого игрока, тип наручников (CuffType), имя надевшего и время, прошедшее с момента надевания.
2. ЕСЛИ в данный момент нет игроков в наручниках, МОД ДОЛЖЕН отобразить сообщение "Нет игроков в наручниках."
3. ЕСЛИ игрок без пермишона `cuffsx.list` выполняет `/cuffsx list`, МОД ДОЛЖЕН отклонить команду.

---

### Требование 9: Команды `/cuffsx enable` и `/cuffsx disable`

**Пользовательская история:** Как администратор сервера, я хочу глобально включать или отключать возможность надевать наручники, чтобы контролировать, когда механика активна.

#### Критерии приёмки

1. КОГДА игрок с пермишоном `cuffsx.enable` выполняет `/cuffsx enable`, Validator ДОЛЖЕН установить глобальный флаг enabled в `true` и отправить подтверждающее сообщение.
2. КОГДА игрок с пермишоном `cuffsx.disable` выполняет `/cuffsx disable`, Validator ДОЛЖЕН установить глобальный флаг enabled в `false` и отправить подтверждающее сообщение.
3. ПОКА глобальный флаг Validator равен `false`, МОД ДОЛЖЕН запрещать все новые надевания наручников (см. Требование 2, критерий 7).
4. ПОКА глобальный флаг Validator равен `false`, МОД ДОЛЖЕН по-прежнему разрешать снятие наручников.
5. ЕСЛИ игрок без необходимого пермишона выполняет `/cuffsx enable` или `/cuffsx disable`, МОД ДОЛЖЕН отклонить команду.

---

### Требование 10: Команда `/cuffsx log`

**Пользовательская история:** Как администратор сервера, я хочу видеть журнал всех взаимодействий с наручниками за последние 2 часа, чтобы проводить аудит активности на сервере.

#### Критерии приёмки

1. КОГДА игрок с пермишоном `cuffsx.log` выполняет `/cuffsx log`, МОД ДОЛЖЕН отобразить все CuffRecord, созданные или удалённые за последние 2 часа, включая: имя целевого игрока, имя надевшего, тип наручников (CuffType), действие (надеты/сняты), блочные координаты и метку времени.
2. ЕСЛИ за последние 2 часа не было взаимодействий, МОД ДОЛЖЕН отобразить сообщение "Нет взаимодействий за последние 2 часа."
3. ЕСЛИ игрок без пермишона `cuffsx.log` выполняет `/cuffsx log`, МОД ДОЛЖЕН отклонить команду.
4. МОД ДОЛЖЕН хранить записи журнала не менее 2 часов после произошедшего взаимодействия.

---

### Требование 11: Интеграция с LuckPerms / Fabric Permissions API

**Пользовательская история:** Как администратор сервера, я хочу, чтобы все команды были защищены пермишонами LuckPerms, чтобы только авторизованный персонал мог их использовать.

#### Критерии приёмки

1. МОД ДОЛЖЕН проверять пермишоны через Fabric Permissions API (`me.lucko.fabric:fabric-permissions-api`) для всех подкоманд `/cuffsx`.
2. МОД ДОЛЖЕН использовать следующие узлы пермишонов: `cuffsx.reload`, `cuffsx.list`, `cuffsx.enable`, `cuffsx.disable`, `cuffsx.log`.
3. ЕСЛИ у игрока нет необходимого узла пермишона, МОД ДОЛЖЕН отправить сообщение "У вас нет прав для выполнения этой команды." и отменить команду.

---

### Требование 12: Серверная совместимость и сборка

**Пользовательская история:** Как разработчик, я хочу, чтобы мод компилировался в рабочий .jar, который можно развернуть на Fabric 1.20.1 сервере.

#### Критерии приёмки

1. МОД ДОЛЖЕН компилироваться без ошибок командой `./gradlew build` с существующими `build.gradle` и `gradle.properties`.
2. МОД ДОЛЖЕН корректно работать в среде выделенного сервера (environment `"*"` в `fabric.mod.json`).
3. МОД ДОЛЖЕН требовать установки на стороне клиента для применения серверных ограничений.
4. МОД ДОЛЖЕН включать все необходимые ресурсные файлы: `fabric.mod.json`, модели предметов, lang-файлы и текстуры-заглушки.
